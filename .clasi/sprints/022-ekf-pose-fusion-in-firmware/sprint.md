# Sprint 022: EKF Pose Fusion in Firmware

## Goals

Replace the fixed-alpha complementary filter in `Odometry::correct()` with a
3-state Extended Kalman Filter that automatically adapts its correction gain
based on accumulated uncertainty. The EKF fuses encoder dead-reckoning
(predict step) with OTOS position observations (update step) at the same
~100 ms cadence as the existing complementary filter.

## Problem

The current `Odometry::correct()` applies a fixed `alphaPos = 0.15` blend at
every OTOS tick, regardless of how much the encoder pose has drifted since the
last fix. This means the filter under-corrects right after a turn (high drift,
but the gain is the same) and over-corrects when the robot has been stationary
(low drift, but the same gain is applied). The demo notebook (sprint 021)
quantified the cost: complementary filter achieves ~32 mm RMS cross-track
error on a figure-eight; a Python-side EKF achieves ~19 mm on the same path
using identical sensor data.

## Solution

Implement a 3-state EKF (`x = [x_mm, y_mm, theta_rad]`) as a self-contained
`EKF` class in `source/control/`. Wire it into `Odometry`: the existing
`predict()` path calls `_ekf.predict()` after computing midpoint kinematics;
a new `correctEKF()` method calls `_ekf.update()` with the OTOS x/y position.
`Robot::otosCorrect()` switches from `odometry.correct()` to
`odometry.correctEKF()`. Three new config fields (`ekfQxy`, `ekfQtheta`,
`ekfROtosXy`) carry the tuned noise parameters into the firmware.

No external math library is used — all 3×3/3×2 matrix operations are unrolled
as plain float arithmetic. OTOS heading is not fused (heading-only EKF channel
is out of scope for this sprint).

## Scope

### In Scope

- `source/control/EKF.h` and `source/control/EKF.cpp` — new 3-state EKF class
- `source/types/Config.h` — three new EKF noise config fields
- `scripts/gen_default_config.py` — EKF defaults added to template
- `source/robot/DefaultConfig.cpp` — regenerated by the script
- `source/control/Odometry.h/.cpp` — `initEKF()`, `correctEKF()`, EKF
  member, `setPose()`/`zero()` EKF sync
- `source/robot/Robot.cpp` — swap `correct()` to `correctEKF()`; wire EKF init
- `tests/dev/test_ekf.py` — pure-Python EKF mirror and unit tests

### Out of Scope

- OTOS heading fusion in the EKF update step (follow-on sprint)
- Camera-fix injection (delayed high-accuracy update)
- SET/GET keys for EKF noise parameters
- Removing the legacy `Odometry::correct()` method (kept for existing tests)

## Test Strategy

T005 (unit tests) covers correctness of the Python-mirrored EKF math, including
predict/update mechanics, covariance growth/shrinkage, convergence, and heading
wrap. The full pytest suite catches regressions in existing odometry, OTOS
fusion, and mock hardware tests. Bench verification via `rogo` commands confirms
the firmware pose is stable under real driving.

## Architecture Notes

The EKF is placed in `source/control/` alongside `Odometry`, which is its only
caller. `Odometry` owns the `EKF` as a value member, avoiding heap allocation on
the embedded target. The EKF is initialised with noise parameters at robot start
via `Odometry::initEKF()`, called from the `Robot` constructor. The existing
`correct()` method is preserved (it is exercised by `test_otos_fusion.py`);
`correctEKF()` is the new OTOS integration path.
