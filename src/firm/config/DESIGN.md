## 1. Purpose

`Config` is the firmware's boot-configuration seam: a small set of
`default*Config()` functions that hand `main.cpp` the per-motor, drivetrain,
OTOS, and planner defaults to construct the robot's hardware objects with.
The seam exists so that per-robot calibration â€” trackwidth, per-port forward
sign, encoder travel calibration, OTOS lever-arm offset, velocity-loop and
heading-loop gains â€” never gets typed into `main.cpp` by hand. It owns
exactly one problem: turning "which physical robot is this" into boot-time
constants, once, before `RobotLoop::run()` starts. It owns nothing about how
those values are used afterward (that's `devices/` and `app/`) or how they
can be changed live post-boot (that's the `DEV M <n> CFG` / `DEV DT CFG` wire
surface â€” see root `DESIGN.md` Â§4 and `docs/protocol-v2.md`).

## 2. Orientation

Two files, two very different authorship models:

- **`boot_config.h`** is hand-written. It declares the `Config::default*()`
  functions, the `OtosBootConfig` struct, and `kMotorConfigCount`. This is
  the stable API `main.cpp` calls.
- **`boot_config.cpp`** is auto-generated by `scripts/gen_boot_config.py`,
  run by `build.py`'s codegen step before every firmware build. The
  generator reads the active robot's JSON config â€” `data/robots/
  active_robot.json` (itself normally just a `{"path": ...}` pointer, e.g.
  to `data/robots/tovez_nocal.json`) or the `ROBOT_CONFIG` environment
  variable override â€” and emits literal `set*()` calls with the resolved
  values baked in as float/int constants, plus a comment naming the source
  JSON path. If no robot config is found (bad path, missing file), the
  generator does not fail the build: every mapping falls back to a firmware
  default baked into `gen_boot_config.py` itself, so the build always
  succeeds, just with bench-placeholder calibration.

Each JSON key maps to at most one baked field; anything the JSON omits falls
back to a firmware default. Some fields (the velocity PID gains' EMA
coefficient, the per-port I2C poll-schedule membership, the drive-pair port
binding) are *never* sourced from JSON at all â€” they're firmware-scheduling
facts or plant-scale-sensitive values that would silently miscalibrate the
robot if a JSON author fat-fingered a stale scale into them (see
`gen_boot_config.py`'s module docstring for the full baked-vs-firmware-default
table). `main.cpp` calls the four `Config::default*()` functions once at
boot, converts the wire-plane `msg::MotorConfig`/`msg::DrivetrainConfig`
results into `Devices::MotorConfig` (the one place both types are visible â€”
see root `DESIGN.md` Â§3, devices isolation invariant), and never touches
`data/robots/*.json` or `gen_boot_config.py` itself again.

## 3. Constraints and Invariants

- **Never hand-edit `boot_config.cpp`:** it is regenerated from
  `data/robots/active_robot.json` (or `ROBOT_CONFIG`) by
  `scripts/gen_boot_config.py` before every build. A hand edit is silently
  destroyed by the next build and gives no error â€” the diagnosis wastes
  real time. Fix the robot JSON or the generator, never the generated file.
- **Calibration is never hardcoded in `main.cpp`:** `main.cpp` calls
  `Config::default*()` and nothing else for boot calibration. Any PR that
  adds a literal calibration constant to `main.cpp` instead of routing it
  through the robot JSON â†’ generator â†’ `boot_config.cpp` path defeats the
  entire point of this subsystem (per-robot builds from one source tree)
  and must be redirected here instead.
- **`port` is a 1-based wire label, not an index:** `msg::MotorConfig::port`
  (and `msg::DrivetrainConfig::leftPort`/`rightPort`) are 1-based brick
  labels â€” a wire/serialized key excluded from the no-units/naming
  conventions' renaming, unchanged since before this subsystem existed.
  `defaultMotorConfigs()` fills `out[i]` 0-based but sets `out[i].port =
  i + 1`; `main.cpp` is the one place that maps the 1-based port back to a
  0-based config-array index. Do not "simplify" this to a 0-based port â€”
  every wire consumer (`DEV M <n> CFG`, telemetry) expects the 1-based
  label.
- **`kMotorConfigCount` must track `Subsystems::NezhaHardware::kMotorCount`:**
  `main.cpp` `static_assert`s the two agree. A motor-count change that edits
  one without the other fails the build loudly (by design) rather than
  silently writing past the end of a 4-entry array; re-run
  `gen_boot_config.py` and update `kMotorConfigCount` together.
- **Missing/bad robot JSON degrades to bench defaults, not a build failure:**
  `gen_boot_config.py`'s `load_robot_config()` prints a warning to stderr and
  falls back to firmware defaults rather than aborting codegen. This is
  deliberate â€” a broken `active_robot.json` must not brick the build â€” but
  it means a silently-wrong calibration (bench placeholders on what should
  be a calibrated robot) looks identical to a clean build. Check the
  generated file's `// Source:` comment line when calibration looks off.
- **`OtosBootConfig` itself is still boot-time-baked only** â€” the BOOT
  defaults (baked from the robot JSON's `geometry.odometry_offset_mm`/
  `calibration.otos_*`) are still consumed exactly once, at `Devices::Otos`
  construction/`begin()`, in `main.cpp`; there is no live SET path for
  `OtosBootConfig` itself and none is planned. **109-004 added a SEPARATE,
  live runtime OVERRIDE** on top of that boot baking: `OtosConfigPatch`
  (`config.proto`/`envelope.proto`), applied by `RobotLoop::handleConfig`
  directly against `Devices::Otos`'s existing `setLinearScalar()`/
  `setAngularScalar()`/`setOffset()`/`init()` primitives (`OL`/`OA`/`OI`,
  host-side `NezhaProtocol.otos_config()`) â€” a live wire-triggered
  RE-calibration after boot, not a rebaking of the boot config struct
  itself. `Otos::setLinearScalar()`/`setAngularScalar()` still write the
  chip's own raw int8 register scalar directly (not the `OtosBootConfig`
  multiplier `begin()` converts once) â€” see `otos.h`'s own OL/OA doc
  comment.

## 4. Design

**Why build-time baking, not a runtime config store.** The target has no
filesystem and no EEPROM story on the nRF52833 â€” there is nowhere on-device
to persist a JSON blob or a key/value store across power cycles. The only
two places calibration can live are "compiled into the image" or "pushed
over the wire after boot and held in RAM until the next reset." This
subsystem owns the first; the second is the existing `DEV M <n> CFG` /
`DEV DT CFG` live-correction surface described in the root architecture
doc. Baking at build time means a single source tree can produce correctly
calibrated images for multiple physical robots (`tovez`, `tovez_nocal`,
`togov`, â€¦) just by pointing `active_robot.json` (or `ROBOT_CONFIG`) at a
different file â€” no runtime robot-identity handshake, no on-device storage
engine, and no risk of a half-written config surviving a mid-flash power
loss.

**Generator shape.** `gen_boot_config.py` is a plain code-templating script,
not a build-system plugin: `load_robot_config()` resolves the active JSON
(env override first, then `active_robot.json`, following a `path` indirection
one level if present), each `*_for_config()`/`*_for_ports()` helper maps a
JSON key (or an unconditional firmware constant, for scheduling facts) to a
value, and `generate()` renders one literal C++ function body per
`Config::default*()` declaration, string-templated with `_f()` formatting
each float as a C++ literal. The `// AUTO-GENERATED` banner and the
`// Source: <path>` comment are load-bearing: they are the only place a
reader can tell which JSON produced a given build without re-running the
generator.

**Sibling generator, deliberately separate.** `scripts/gen_default_config.py`
bakes the *old* `source/robot/RobotConfig` struct for the legacy tree; it is
not this generator and does not share code with it, because the two target
different C++ types on different velocity-PID plant scales (the old
`RobotConfig` loop is tuned in PWM-percent units, `~kp 0.3`; the current
`NezhaMotor` loop operates on a duty `[-1,1]` scale, `~kp 0.002`â€” see the
robot JSON's `control._vel_gains_domain` marker). Merging the two generators
would risk silently applying one plant's gains to the other's loop.

**Fallback discipline is uniform.** Every mapping in `gen_boot_config.py`
follows the same shape: read a JSON key if present, else use a named
`_DEFAULT` constant declared at module scope. This makes "what does an
unmigrated or minimal robot JSON boot with" answerable by reading the
constants block once, rather than re-deriving it per field.

## 5. Interfaces

### Exposes
- **`Config::defaultMotorConfigs(msg::MotorConfig* out)`:** fills
  `out[0..kMotorConfigCount-1]` with per-port boot defaults (velocity gains,
  forward sign, travel calibration, EMA filter coefficient, poll-schedule
  membership). Caller-owned buffer, exactly `kMotorConfigCount` entries;
  called once, at boot, from `main.cpp`.
- **`Config::defaultDrivetrainConfig() -> msg::DrivetrainConfig`:**
  trackwidth and the drive-pair port binding (1-based labels).
- **`Config::defaultOtosBootConfig() -> Config::OtosBootConfig`:** OTOS
  lever-arm mounting offset (x/y/yaw) and linear/angular scale multipliers.
  Consumed directly by `main.cpp`'s `Hal::OtosOdometer` construction â€” never
  a wire surface (Â§3).
- **`Config::defaultPlannerConfig() -> msg::PlannerConfig`:** the full
  motion-limit/tracker/policy field set (acceleration/velocity/jerk
  ceilings, outer heading-loop PD gains, `Drive::Limits` fields) baked from
  the robot JSON's `control.*` keys where present.

### Consumes
- **`msg::MotorConfig`/`msg::DrivetrainConfig`/`msg::PlannerConfig`** (from
  `messages/`) â€” the wire-plane structs this subsystem's functions return;
  see [messages/DESIGN.md](../messages/DESIGN.md).
- **`data/robots/*.json`** (build-time only, via the generator, not at
  runtime) â€” the per-robot calibration source; schema at
  `data/robots/robot_config.schema.json`.
- **`scripts/gen_boot_config.py`** (build-time codegen, invoked by
  `build.py`) â€” produces `boot_config.cpp` from the above.

## 6. Open Questions / Known Limitations

- Several `PlannerConfig` fields (`arrive_tol`, `turn_in_place_gate`) are
  left permanently unset (0.0f) because nothing reads them yet â€” carried
  forward unchanged from the pre-generator `main.cpp` defaults. Revisit if a
  consumer ever needs them.
- `togov` (mecanum) currently inherits the same firmware-default
  `Drive::Limits` fields as an "uncharacterized robot" rather than having its
  own bench-measured values (see root `architecture-update.md` Open Question
  6) â€” `tovez.json`'s `_drive_limits_note` documents the per-field derivation
  for the one robot that has been characterized.
- There is no build-time check that a robot JSON's `control._vel_gains_domain`
  marker actually matches the current plant scale; an unmigrated JSON that
  claims the new domain but carries old-scale numbers would silently
  miscalibrate the velocity loop. Caught today only by bench verification,
  not by codegen.
