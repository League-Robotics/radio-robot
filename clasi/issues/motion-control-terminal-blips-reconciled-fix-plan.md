# Motion-control terminal blips — reconciled fix plan

## Why this issue exists

Two independent reviews landed on 2026-07-18 about the same symptom — the
D700 straight shows a ~44 mm/s velocity hump *after* the move should have
finished, and the 360° pivot shows a ±15 mm/s sign-changing tail. This issue
reconciles them into one plan. The two reports are:

- **Simplification review** (`…-simplification-review.md`) — "Why the Blips
  Exist and What to Delete."
- **Architecture review** (`motion-control-architecture-review.docx`) —
  "Motion Control Architecture Review," findings F1–F7.

Both are read-only; neither modified the tree. This issue is the **plan-first
artifact** — no implementation is authorized by filing it. The work is a
multi-sprint arc and should be roadmap-planned up front.

---

## 1. Where the two reports agree (the diagnosis is not in dispute)

Independently, both reviews reached the same core conclusions. Treat these as
settled:

1. **The blips are software-synthesized, not sim noise or physics.** The
   second lobe / terminal tail is present in the *commanded* trace, so it is
   generated by the controller, not by measurement noise or an ideal-vs-real
   plant argument. Neither "it's a sim, it should be perfect" nor "the blips
   are inevitable physics" is correct.
2. **The Ruckig 1-D scalar solve is sound — keep it.** The trajectory
   generator is not the problem. Everything layered *around* it is.
3. **The same patch stack must be deleted** (both name the same set): straight
   target padding (`kStraightLeadBias/Slope`, `1.5 + 0.102·v_cruise`), the
   terminal top-up mini-profile + 3 mm cross-bias, the pivot overshoot lead
   slope, the `plan_lead` time-shifted sampling, `terminal_lead`,
   `heading_lead_bias`, the minimum-speed heading floor (in ideal motion), and
   the overshoot-carry between commands.
4. **The same 18 dead `PlannerConfig` fields must go** — both lists are
   verbatim identical: `arrive_tol`, `turn_in_place_gate`, `v_wheel_max`,
   `steer_headroom`, `wheel_step_max`, `track_k_s`, `track_k_theta`,
   `track_k_cross`, `trim_v_max`, `trim_omega_max`, `replan_err_pos`,
   `replan_err_theta`, `replan_hold`, `replan_min_period`, `replan_max`,
   `handoff_tol_pos`, `handoff_tol_v`, `arrive_vel_tol`. No live firmware
   consumer; reserve the proto field numbers.
5. **Executor is overloaded** (`executor.cpp` = 1171 lines, `executor.h` =
   637, verified today) and should be decomposed into narrow
   responsibilities: pure trajectory · reference mapper · wheel controller ·
   lifecycle/safety.
6. **Split the simulator into two explicit purposes** — an *ideal* mode
   (τ→~0, deadband/dwell/quantization off) for trajectory & geometry
   acceptance, and the current *hardware-parity* mode (τ=130 ms + full write
   shaping) for controller robustness. In parity mode, do **not** require
   `actual` to overlay `commanded`; require bounded, monotonic, non-ringing
   tracking within a stated tolerance.
7. **Completion means "command zero once at trajectory end."** A measured
   tolerance may *delay the completion event* in parity mode, but it must
   **never authorize a new motion trajectory**. A large endpoint disagreement
   is a tracking failure/fault, not a license to re-solve.
8. **There is a repeated-move stale-state reliability bug** — the windage
   sweep spins up a fresh `SimLoop` per run because accumulated executor state
   reaches solve-failure after ~5 pivots. Expose it with a permanent
   **same-boot regression test**, don't hide it behind test isolation.
9. **Do not promote the turn-windage lookup table into production.** It fits a
   delay artifact; on its own numbers the *compensated* final heading (357.31°)
   is worse than the *baseline* (360.11°) for a 360° request.
10. **Keep the safety machinery**: gross-divergence reanchor (40 mm), the
    timeout backstop, emergency-stop-on-solve-fail, the unwrapped heading
    accumulator (109-009), and hardware-only deadband/reversal protection
    (protects against the documented encoder-wedge fault — not gold-plating).

## 2. The one real divergence — and why it's complementary, not contradictory

The reports differ on *what the missing control law is*. They are two lenses on
the same open-loop problem, and the reconciliation is to do **both**, not pick
one.

| | Simplification review | Architecture review |
|---|---|---|
| Named root cause | Feedforward **without feedback** on the dominant (linear) channel | (F1) wrong cycle order → multi-cycle latency; (F2) lead-sampling time-warps Ruckig, breaking the jerk guarantee; (F5) velocity-only wheel FF omits the acceleration term |
| Proposed cure | Add linear position feedback: `v_cmd = v_ref(t) + k_v·(s_ref − s_meas)`, mirroring the existing heading PD | Fix the cycle schedule, then pass Ruckig acceleration into the wheel controller (`u_ff = (v_ref + τ·a_ref)/K`) + *small* state feedback |
| View of the leads | Open-loop Taylor approximations of `k·(ref − meas)`; delete them, close the loop | Timing patches for a mis-ordered pipeline + wheel lag; remove once the schedule is correct |

**These compose.** Model feedforward (the `τ·a_ref` term) cancels the *nominal*
`v·τ` ramp lag analytically; a bounded position-feedback trim cleans up
*residual / model-mismatch* error and is what keeps hardware honest when τ, gains,
or the plant drift. The architecture review already says "model feedforward +
small state feedback"; the simplification review's own inequality analysis has
feedforward and feedback coexisting. There is no conflict — the leads die in
*both* schools; the disagreement is only over what replaces them, and the answer
is "both a corrected feedforward and a bounded feedback trim."

The architecture review contributes two findings the simplification review
under-weights, both **verified in-source today**:

- **F1's durable concern is real, but read the current tree carefully.** The
  on-disk state the architecture review snapshotted — `drive_.tick()` at the
  *top* of `cycle()` while `pilot_.tick()` runs late in kSettle, contradicting
  the `pilot_.tick()` "BEFORE `drive_.tick()`" comment — is **a deliberate,
  temporary reorder** on the `pid-debugging` branch (Eric hoisted `drive_.tick()`
  to test whether stage ordering contributes to the motion errors; it will be
  reverted). So the comment-vs-code mismatch is **not** itself a standing
  defect and should not be "fixed" as one. What *does* survive the revert is
  the underlying latency: even in the intended schedule, odometry/OTOS integrate
  at kPace (end of cycle) and are read by the *next* cycle's `pilot_.tick()`.
  That ~1-cycle sensor staleness is independent of the drive/pilot swap and is
  independently corroborated by `later/turn-lead-compensation-gain-cotuning.md`
  ("`applyOtosSample()` runs in the kPace block AFTER `Pilot::tick()` reads
  heading"). Evaluate F1 against the *restored* baseline, not this experiment.
- **F2 jerk-warp math is real.** Sampling at `elapsed + lead` during lead
  ramp-in evaluates the reference at `2t`, so `a_cmd = 2·a_plan` and
  `j_cmd = 4·j_plan`, with a discontinuity when the lead stops growing. The
  emitted command is not the trajectory Ruckig certified.

The simplification review contributes one concrete bug the architecture review
only implies:

- **Stale-twist-on-idle.** `pilot.cpp:74-76` calls `drive_.setTwist()` only
  while `state() != kIdle`; on the running→idle transition Drive is never
  zeroed, so the robot creeps at the last staged twist until the 300 ms
  deadman lease (`kPilotDeadmanLease`, robot_loop.cpp:57) expires and calls
  `drive_.stop()`. That is the post-completion shelf. Worth fixing on its own,
  today.

## 3. Reconciled fix plan (dependency order, each step independently verifiable)

The wheel-speed trace + the sim tour gates are the acceptance instrument for
each step. Land a numeric jerk / single-lobe acceptance test **first** so every
subsequent deletion is guarded.

0. **Lock behavior down.** Add end-to-end numeric acceptance tests for the
   D700 straight and 360° pivot that (a) differentiate the emitted wheel
   setpoints and assert velocity/accel/jerk limits at start and stop, (b)
   assert exactly one velocity lobe per wheel (straight) / one +lobe and one
   −lobe (pivot), and (c) fail on any correction lobe, reversal, second solve,
   or solve-fail event. Add a **same-boot** test that runs dozens of
   consecutive straights and pivots to expose stale executor state (§1.8).
1. **Fix the stale-twist-on-idle bug** (simplification review, §2). On the
   running→idle transition, stage `drive_.setTwist(0, 0)` once. Removes the
   post-completion shelf on every terminal command immediately. One line;
   independent of everything else.
2. **Do the F1 cycle-order A/B comparison LAST, just before hardware.** The
   current on-disk reorder (`drive_.tick()` hoisted above `pilot_.tick()`) is a
   live experiment that **already changed the observed errors** — so it stays in
   place through all the other steps below (the current sim tests depend on it),
   and the revert-and-compare is deferred to just before the bench pass. Tracked
   as its own issue: `cycle-order-reorder-experiment-ab-before-hardware.md`.
   The durable, ordering-independent half of F1 — the ~1-cycle odometry/OTOS
   staleness (`applyOtosSample()` at kPace, read next cycle) — can still be
   addressed here by restructuring `cycle()` toward
   `sense → odometry → Pilot → Drive → wheel-write` so a fresh reference reaches
   the wheel PID the same cycle it is produced. **Constraint:** this touches
   `robot_loop.cpp` scheduling, which has real bus-timing implications — the
   0x46 encoder-select latch interleave, the `runAndWait` settle/clearance
   windows (Eric's chosen wait form — keep it), and OTOS I2C tick placement
   (see the 098-004 revert). Reorder within those constraints; do not
   naïvely flatten the schedule.
3. **Delete the lead-sampling machinery (F2 / `plan_lead`).** Sample the
   trajectory at `t`, not `t + lead`. Restores the jerk guarantee.
4. **Add acceleration feedforward through the mapping layer (F5).** Ruckig
   already computes `a_ref` (`jerk_trajectory.cpp`) and it is discarded before
   the wheel layer — pass it through and use `u_ff = (v_ref + τ·a_ref)/K` (or
   the exact discrete inverse). Cancels the nominal ramp lag analytically.
5. **Add a bounded linear position-feedback trim** (simplification review):
   `v_cmd = v_ref(t) + k_v·(s_ref − s_meas)`, clamped to ±v_ceiling;
   `s_meas` = `measuredPathSinceActivation_` (already exists). Keep the
   existing heading PD as the angular tracker. This mops up residual /
   model-mismatch error that pure feedforward leaves.
6. **Delete the terminal patch stack.** Straight-lead padding, terminal
   top-up + cross-bias, pivot overshoot lead, overshoot-carry, the
   minimum-speed floor, and the EMA/leaky-counter dwell machinery. Completion
   becomes: `t ≥ duration + margin` AND `|s_err| < s_tol` AND `|θ_err| < θ_tol`,
   held for one `arrive_dwell`, with a **single** timeout backstop.
7. **Strip the 18 dead `PlannerConfig` fields** end-to-end (proto, generated
   structs, patch-merge in `pilot.h`, wire rows) — reserve field numbers.
   Also drop the now-unused lead fields (`plan_lead`, `terminal_lead`,
   `heading_lead_bias`, `min_speed`), folding OTOS measurement-age projection
   into `HeadingSource` if hardware still wants it.
8. **Split ideal vs. hardware-parity sim** and correct the parity plant's
   Euler position update to the analytical form (F4). Ideal mode gates
   trajectory/geometry acceptance; parity mode gates robustness.
9. **Decompose Executor** into the four responsibilities once the deletions
   have shrunk it.
10. **Only then** tune the small residual physical/hardware-completion
    parameters on the real robot.

Expected net effect (from the reports, to be confirmed): `executor.cpp`
≈ 1171 → ~700 lines, `executor.h` ≈ 637 → ~300, `PlannerConfig` 35 → ~14
fields, motion-path tuned constants ~15 → ~4 (two gains, two tolerances).

## 4. Guardrails — what must NOT regress

- **Do not resurrect the 087-009 / terminal-reversal failure.** Both disasters
  came from **re-solving trajectories from measured state near the target**
  (a time-optimal solve from nonzero velocity to a near-zero remainder is an
  overshoot-and-reverse plan). The new law never solves from measured state:
  JerkTrajectory solves stay plan-state-seeded, one per command. Feedback
  enters only as a **bounded** velocity/omega trim on the sampled reference —
  near the target its authority (`k_v·s_tol`) is below the deadband, incapable
  of the old full-authority reversal. (See memory: "gap first THEN reversals,
  never reintroduce reversals.")
- **Keep the `runAndWait(gap, body)` cycle form** — the wait carries the name,
  the block scopes the borrowed time. Do not re-argue a flat `sleepUntil`.
- **Respect the deadband inequality** the simplification review states: for the
  floor to be *deletable* rather than merely moved, gains × tolerance must
  clear the deadband-equivalent speed (`k_h·θ_tol ≥ ω_deadband`,
  `k_v·s_tol ≥ v_deadband`). At the 3% duty deadband, `heading_kp = 6.0`
  (bench value) clears it at 3° tol; the boot default `1.0` does not. If the
  inequality can't be met on hardware, keep a *small explicit* floor — but as
  a stated constraint, not a discovered workaround. On hardware, stiction
  belongs to the wheel PID's integral term (`vel_ki`), not the trajectory
  layer.
- **Hardware deadband/reversal-dwell stays on real motors** — it guards the
  reproducible encoder-wedge fault.

## 5. Corrections / caveats found while verifying (don't quote the reports blind)

- **The main loop is currently reordered as a live experiment — and it stays
  that way for now.** On `pid-debugging` (as of 2026-07-19) `drive_.tick()` is
  hoisted above `pilot_.tick()`, contradicting the `pilot_.tick()` comment. This
  is Eric's test of whether stage ordering causes the motion errors; it
  **already changed the observed errors**, so it is kept in place through the
  rest of this fix plan (the current tests depend on it). The revert-and-compare
  is deferred to just before hardware — see
  `cycle-order-reorder-experiment-ab-before-hardware.md`. Do not file the
  comment-vs-code mismatch as the F1 defect, and re-read `robot_loop.cpp` before
  quoting cycle order.
- **Cycle time is 20 ms, not 50 ms.** Both reports assume a 50 ms cycle;
  `robot_loop.cpp:25` now has `kCycle = 20` (`kSettle`/`kClear` = 0). The
  multi-cycle latency in F1 is real, but its *magnitude* is ~2×20 ms, not
  ~2×50 ms — re-derive any latency-dependent number before quoting it.
- **This issue supersedes `later/turn-lead-compensation-gain-cotuning.md`.**
  That issue (109-010 follow-up) proposes **co-tuning** the three leads
  (`heading_lead_bias`, `plan_lead`, `terminal_lead`). The reconciled direction
  is to **delete** those leads and close the loop instead. Fold that issue into
  this arc; do not pursue lead co-tuning in parallel.
- The `motor_armor.h:16` comment claiming the sim composes bare motors is stale
  — `sim_harness.h` wraps them in armor since the PARITY change. Fix the
  comment when the area is touched.

## 6. Merged acceptance criteria

**700 mm straight:** exactly one velocity lobe per wheel; no nonzero command
after the first terminal zero; no reversal/top-up/second solve; emitted accel &
jerk within bounds incl. start/stop; final wheel position = 700 mm within
declared tolerance; final wheel velocity and acceleration = 0.

**360° pivot:** wheel distances = ±π·trackWidth; heading = 2π within tolerance;
one +lobe and one −lobe; no wheel sign change after terminal deceleration
begins; no heading-PD or min-speed tail after the planned endpoint in ideal
mode.

**Cross-cutting:** numerically differentiate emitted setpoints and assert
v/a/j limits (not just "no one-tick jump"); test pure-trajectory, ideal-actuator,
and parity-actuator as separate layers; run dozens of consecutive moves in one
boot; record requested / planned / measured endpoint + completion reason as
separate telemetry; tours stay 100% reliable; full suite green.
