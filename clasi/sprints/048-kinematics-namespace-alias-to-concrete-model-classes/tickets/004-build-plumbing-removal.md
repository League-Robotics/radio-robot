## Description

Remove the `ROBOT_DRIVETRAIN_MECANUM` macro from the build system entirely. This
is the ticket that makes `grep -rn ROBOT_DRIVETRAIN_MECANUM CMakeLists.txt
tests/_infra/sim/CMakeLists.txt build.py` return zero.

**This ticket MUST run after tickets 001, 002, and 003.** All `#ifdef
ROBOT_DRIVETRAIN_MECANUM` branches in source files must already be stripped before
the macro definition is removed from CMake. If any source `#ifdef` remained, dropping
the macro definition could silently change which code path is compiled (the `#ifdef`
body would now evaluate false, potentially leaving a different body active than before).

After this ticket: the build never defines `ROBOT_DRIVETRAIN_MECANUM`, never reads
`drivetrain_type` from the robot JSON, and never passes `-DROBOT_DRIVETRAIN=` to the
compiler. `MecanumHAL.cpp` is explicitly excluded. `MecanumKinematics.cpp` is
explicitly retained.

## Implementation Plan

### Approach

Three-file edit. Start with `CMakeLists.txt` (root), then
`tests/_infra/sim/CMakeLists.txt`, then `build.py`. Each can be verified
independently via CMake configure + a grep check before moving on.

### Files to Modify

**`CMakeLists.txt`** (root, ~302–329)

Locate the `ROBOT_DRIVETRAIN` block. It likely reads something like:

```cmake
# Read drivetrain type from robot JSON
if(DEFINED ROBOT_JSON)
    # ... parse drivetrain_type ...
    set(ROBOT_DRIVETRAIN ...)
endif()
if(ROBOT_DRIVETRAIN STREQUAL "mecanum")
    add_definitions(-DROBOT_DRIVETRAIN_MECANUM)
endif()
```

Delete this entire block. Then locate the `SOURCES` list and:
- Explicitly exclude `source/io/real/MecanumHAL.cpp` with a `list(REMOVE_ITEM
  SOURCES ...)` or by not including it in the glob, plus a comment.
- Confirm `source/kinematics/MecanumKinematics.cpp` remains included (it is pure
  math, no `#ifdef`, already compiled in the host/sim build; confirm it is not
  firmware-excluded).

**`tests/_infra/sim/CMakeLists.txt`** (~32–43)

Delete:
- Any `add_definitions(-DROBOT_DRIVETRAIN_MECANUM)` line.
- The dual-config block that set up `build_mecanum/` as an alternate output
  directory or a second CMake configuration.
- Any `if(MECANUM) ... endif()` blocks.

**`build.py`**

- Delete the `_read_drivetrain_type()` function body (lines 79–103). If it is
  called from multiple sites, remove all call sites first.
- Delete the `-DROBOT_DRIVETRAIN=...` CMake argument passing at lines 197–201 and
  229. These were something like:
  ```python
  drivetrain = _read_drivetrain_type()
  cmake_args += [f"-DROBOT_DRIVETRAIN={drivetrain}"]
  ```
  Remove both the variable assignment and the `cmake_args` append.
- Verify no remaining references to `ROBOT_DRIVETRAIN` or `drivetrain_type` in
  `build.py`.

### Testing Plan

1. `grep -rn ROBOT_DRIVETRAIN_MECANUM source tests CMakeLists.txt build.py` → zero.
2. `python build.py --clean` → clean compile, no warnings.
3. Decode `MICROBIT.hex` to confirm it is a live build (not stale incremental).
   Per the knowledge note: `build.py --clean` forces a full rebuild; verify the
   hex contains a recent timestamp or use the decode step documented in the
   stale-incremental-build knowledge entry.
4. `uv run pytest` → all green.
