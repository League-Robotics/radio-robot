# Sprint 111: Motion-control terminal blips: close the loop and delete the compensator stack

## Goals

This is **arc sprint 1 of a multi-sprint motion-control arc** driven by
`clasi/issues/motion-control-terminal-blips-reconciled-fix-plan.md` (the
"reconciled fix plan" — a merge of two independent 2026-07-18 reviews of the
D700-straight terminal hump and 360° pivot terminal tail). The reconciled
plan lays out a 10-step dependency-ordered execution plan (steps 0-10); this
sprint scopes a coherent, low-to-moderate-risk, green-closeable slice of it:

- **Step 0** — a numeric behavior-lock acceptance harness that documents
  today's known-bad shape (the hump/tail) as an explicit, machine-checked
  regression fence, so every later deletion in this arc is guarded.
- **Step 1** — the stale-twist-on-idle fix (`App::Pilot::tick()`), which
  removes the post-completion "shelf" (the robot creeping at the last
  staged twist until the 300ms deadman lease expires) on its own, today,
  independent of everything else in the arc.
- **Step 7** — strip the 18 dead `PlannerConfig`/`PlannerConfigPatch`
  fields end to end (proto, generated structs, `Pilot::applyPlannerPatch`
  merge, wire), reserving their field numbers. Mechanical, low behavioral
  risk, verified dead by grep before removal.
- **Baseline triage** — master currently carries 4 failing sim tests,
  introduced by already-merged `pid-debugging` WIP (a motor-interface
  refactor + a sim-behavior change + an in-place cycle-order/request-collect
  reorder in `robot_loop.cpp`, all three named in the driving issue's own
  corrections section). `close_sprint` runs `uv run pytest` — this sprint
  gets the suite green (fix or quarantine) so this sprint and the rest of
  the arc can close cleanly.

The risky heart of the arc — deleting the lead-sampling machinery, adding
acceleration feedforward, adding a bounded linear position-feedback trim,
and deleting the terminal patch stack (the issue's steps 2-6) — is
deliberately **out of scope** for this sprint. See "Out of Scope" below and
the Architecture section's Design Rationale for why.

## Problem

The D700 straight shows a ~44mm/s velocity hump *after* the move should
have finished; the 360° pivot shows a ±15mm/s sign-changing terminal tail.
Both reviews agree the blips are software-synthesized (present in the
*commanded* trace, not sensor noise) and are generated by a patch stack of
open-loop lead-compensation and terminal top-up logic layered around an
otherwise-sound Ruckig jerk-limited solve. Fixing this safely requires a
numeric acceptance instrument to exist *before* any deletion — otherwise
"the hump is gone" cannot be distinguished from "the hump moved" or "a new
regression was introduced" (the arc's own explicit history: bug 087-009,
a terminal-reversal failure caused by re-solving from measured state, is
the standing cautionary example of an ungated motion-control change).

Separately, one concrete, independently-fixable bug (stale-twist-on-idle)
and one purely mechanical cleanup (18 dead config fields) are ready to land
now without waiting for the harness's own later assertions to go green, and
a pre-existing, unrelated test-suite regression (4 failing sim tests from
already-merged WIP) blocks this sprint — and the rest of the arc — from
closing at all.

## Solution

Four sequenced tickets, each independently leaving `uv run pytest` green:

1. Build a behavior-lock harness (`src/tests/sim/system/`) that drives a
   D700 straight and a 360° pivot through the REAL `App::RobotLoop` /
   `App::Pilot` / `Motion::Executor` stack (via `TestSim::SimHarness`,
   the same compiled-per-test-binary pattern every sibling `system/*.py`
   test already uses), captures the per-cycle wheel-velocity trace,
   numerically differentiates it for accel/jerk, and asserts limits and
   single-lobe shape — marking every assertion the CURRENT patch stack
   cannot yet satisfy `xfail(strict=False)` with a comment citing the
   driving issue, plus a same-boot repeated-move reliability scenario.
2. Triage and green the 4 pre-existing sim-test failures: fix what is
   genuinely broken and independent of the in-place `robot_loop.cpp`
   reorder; quarantine (xfail, with a comment citing the deferred
   cycle-order issue) only what the investigation actually traces to that
   reorder.
3. Fix stale-twist-on-idle in `App::Pilot::tick()`.
4. Strip the 18 dead `PlannerConfig`/`PlannerConfigPatch` fields through
   `protos/` + the generator (never a hand-edit to generated code).

## Architecture

**Sizing: Substantial** — this sprint touches 3+ existing modules
(`App::Pilot`, the `messages/` wire schema + generator output, and the
`tests/sim` harness/test-infrastructure tree) and adds one new test module
(the behavior-lock harness). No new production cross-module dependency, no
dependency-direction change, and no production data-model change (the
config-schema edit is a shrink of dead fields, not a new shape) — but the
module count alone crosses the "3+ modules touched" substantial-tier
signal, so this uses the full write-up rather than the compact variant.

### Architecture Overview

**No diagram.** Every module this sprint touches (`App::Pilot`, the
`messages/` schema + generator output, three existing sim-test files) is
pre-existing, and this sprint changes none of their edges to one another —
`Pilot`'s dependency on `Motion::Executor`/`App::Drive`, `messages/`'s
position as a dependency-free leaf, and the `tests/sim` harness's
dependency on the real `App::RobotLoop` graph are all unchanged. The one
new module (the behavior-lock harness) is a read-only *observer* wired
onto the exact same `TestSim::SimHarness` composition every sibling
`system/*.py` test already uses (`move_queue_harness.cpp`,
`heading_source_harness.cpp`) — it adds no new edge a diagram would need to
show, only a new leaf hanging off an already-diagrammed point. This mirrors
sprint 020's own precedent for stating "no diagram" within the substantial
tier: independent, non-interacting fixes across existing modules, not a new
subsystem or a new composition between modules.

**What Changed**

- **`App::Pilot::tick()`** (`src/firm/app/pilot.cpp`) gains a
  before/after snapshot of `executor_.state()` around the
  `executor_.tick()` call. Today's code only stages a twist while
  `state() != kIdle`, so a command that completes naturally *inside* a
  `tick()` call (state was non-idle before, `kIdle` after) leaves `Drive`
  holding whatever twist was staged the PREVIOUS cycle, forever, until the
  300ms deadman lease expires and force-stops it — the "post-completion
  shelf." The fix stages exactly one `setTwist(0, 0)` on that specific
  transition. A flush-caused idle transition (TWIST/STOP preemption) is
  unaffected: `processMessage()` calls `Pilot::flush()` earlier in the SAME
  cycle, before `Pilot::tick()` runs, so `stateBefore` is already `kIdle`
  by the time `tick()` samples it, and the "just completed" branch is
  never taken — the raw command's own twist (already staged by
  `handleTwist()`) is preserved untouched.
- **`msg::PlannerConfig`/`msg::PlannerConfigPatch`** (`src/protos/
  planner.proto`, `src/protos/config.proto`) lose 18 fields with no live
  firmware consumer (verified by grep across `src/firm/` before removal —
  see Migration Concerns): `arrive_tol`, `turn_in_place_gate` (declared on
  `PlannerConfig` only, never in the patch — `boot_config.h`'s own comment
  already documents them as permanently unset with no consumer), plus 16
  fields declared on BOTH `PlannerConfig` and `PlannerConfigPatch`:
  `v_wheel_max`, `steer_headroom`, `wheel_step_max`, `track_k_s`,
  `track_k_theta`, `track_k_cross`, `trim_v_max`, `trim_omega_max`,
  `replan_err_pos`, `replan_err_theta`, `replan_hold`, `replan_min_period`,
  `replan_max`, `handoff_tol_pos`, `handoff_tol_v`, `arrive_vel_tol`. Their
  field numbers are `reserved` in both proto messages. `App::Pilot::
  applyPlannerPatch()`'s 16 corresponding merge lines are deleted.
  Generated output (`messages/planner.h`, `messages/config.h`,
  `messages/wire.{h,cpp}`, `messages/layout_checks.{h,cpp}`) is
  regenerated via `scripts/gen_messages.py`, never hand-edited (the
  `messages/` subsystem's own invariant). `PlannerConfig`'s live fields
  (`a_max`/`a_decel`/`v_body_max`/`yaw_rate_max`/`yaw_acc_max`/`j_max`/
  `yaw_jerk_max`, `min_speed`, `heading_kp`/`heading_kd`, `arrive_dwell`,
  `heading_source`, `heading_dwell_tol`/`heading_dwell_rate`,
  `heading_lead_bias`/`plan_lead`/`terminal_lead`) are untouched — this is
  a pure removal, not a renumbering of anything that survives.
- **A new behavior-lock harness module** under `src/tests/sim/system/`
  (a new `*_harness.cpp` + `test_*.py` pair, following the exact
  established shape of `move_queue_harness.cpp`/`test_move_queue.py`):
  commands a D700 `kArc` straight and a 360° `kPivot` via
  `TestSim::SimHarness::injectMove()`, captures each cycle's decoded
  `Telemetry::Frame.velLeft`/`velRight` into a trace, numerically
  differentiates it in the harness itself for accel/jerk, and asserts
  bounds/lobe-count/"no command survives the terminal zero" — printing a
  human-readable trace and exiting 0/1 like every sibling harness. A
  companion same-boot scenario reuses ONE `SimHarness` instance (no
  fresh-boot-per-run, unlike `turn_windage_sweep.py`'s own deliberate
  per-run isolation) to run dozens of consecutive moves and assert none
  faults, targeting the §1.8/F7 stale-executor-state reliability bug.
- **Three existing sim-test files** (`test_plant.py`, `test_sim_api.py`,
  `test_app_robot_loop.py`, plus `test_profiled_motion_sim.py`'s turn
  scenario) get either a root-cause fix or a cited `xfail`/`skip`
  quarantine — see ticket 002's own acceptance criteria for the
  per-test disposition this sprint's own investigation already found (this
  Architecture section's Migration Concerns subsection below records the
  findings; ticket 002 verifies and acts on them).

**Why**

The behavior-lock harness exists so every subsequent arc sprint's
deletions are checked against a real, numeric, automated fence instead of
an eyeballed trace — the arc's own stated method ("land a numeric jerk /
single-lobe acceptance test first so every subsequent deletion is
guarded"). The stale-twist-on-idle fix and the dead-field removal are
pulled forward into this same sprint because both are self-contained,
independently verifiable, and unblock real value (a visibly better robot,
a smaller/clearer config surface) without waiting on the arc's riskier
control-law work. The baseline triage exists because `close_sprint` runs
the full suite — a red suite this sprint did not cause still blocks this
sprint (and the rest of the arc) from closing, so it must be resolved now,
not deferred.

**Impact on Existing Components**

- `App::Pilot`: behavior change (Step 1) — a Move that completes naturally
  now visibly stops the robot instead of creeping for up to 300ms.
  `Motion::Executor`, `App::Drive`, `App::HeadingSource` are unchanged.
- `messages/` wire schema: shrinks (Step 7) — any host-side code that
  still SETS one of the 18 removed `PlannerConfigPatch` fields will find
  the field absent after this sprint (see Migration Concerns — verified no
  such caller exists today).
- `src/tests/sim/system/`: gains one new harness module; three existing
  harnesses/tests get a fix or a quarantine, per ticket 002.
- `Motion::Executor`, `Motion::JerkTrajectory`, `robot_loop.cpp`'s cycle
  schedule: **untouched** by every ticket in this sprint.

### Design Rationale

**Decision 1 — defer the straight-only linear-loop-closure opt-in offered
to this sprint.**
- *Context*: the dispatching brief explicitly permitted including the
  D700-straight-only slice of steps 3+5+6 (delete the straight-lead
  padding + terminal top-up, add a bounded linear position-feedback trim
  for the `kArc` linear channel) in this sprint IF it is "feasible AND
  fully verifiable behind the new harness within this sprint's risk
  budget."
- *Alternatives considered*: (a) defer the entire control-law surgery to
  sprint 2 (chosen); (b) include the straight-only slice as a final ticket
  in this sprint, gated by ticket 001's harness.
- *Why this choice*: the harness ticket 001 builds has zero track record
  the moment it would need to gate a real control-law change in the SAME
  sprint — there is no prior sprint's worth of evidence that its
  assertions correctly discriminate a real fix from a subtle regression.
  `executor.cpp` is ~1171 lines with several interacting terminal-decision
  mechanisms (the dwell gate, the `STOP_TIME` backstop, the leaky-counter
  hold, the settle epsilon, `pendingLinearRetarget_`'s own terminal
  top-up) that the straight-only slice would need to touch without
  disturbing the pivot path, and the arc's own standing cautionary
  example (bug 087-009, a terminal-reversal failure from re-solving off
  measured state) is exactly the failure class a rushed, undercooked
  control-law change in a debut-harness sprint risks repeating. Stacking
  that risk onto a sprint whose OWN stated sizing target is "low-to-
  moderate" is not justified when steps 3-6 already have a clean, fully
  scoped home in sprint 2.
- *Consequences*: sprint 2 opens with a merged, proven (ticket 001 ran
  green/xfail-as-expected through tickets 002-004's own unrelated changes)
  harness and a green baseline, and can spend its entire budget on the
  control-law deletion+addition with real regression coverage from its
  first ticket. The D700 straight's hump therefore is NOT fixed by this
  sprint — only documented (xfail) and instrumented.

**Decision 2 — dead-field verification is grep-then-remove, not
remove-then-see-what-breaks.**
- *Context*: the driving issue names the 18 fields as dead from its own
  source reading; this sprint independently re-verified deadness rather
  than trusting the issue's list blind.
- *Alternatives considered*: (a) grep `src/firm/` for every field name
  before touching the schema, confirming zero consumers outside the
  generated struct declaration and the `applyPlannerPatch` merge arms
  (chosen, and already performed during this sprint's own planning —
  confirmed a match against the issue's list, including the two
  non-patch fields `arrive_tol`/`turn_in_place_gate`, which
  `config/boot_config.h`'s own comment independently corroborates as
  "unused by any current consumer"); (b) remove first and let the build
  fail loudly if something used a field.
- *Why this choice*: (b) risks a *silent* behavior change if a field were
  read somewhere the compiler wouldn't flag (e.g. a default-initialized
  struct field read by value with no explicit setter call) — (a) is a
  stronger, pre-verified guarantee of true deadness, matching the sprint
  brief's own explicit instruction to verify by grep first.
- *Consequences*: ticket 004 documents the grep evidence in its own
  acceptance criteria rather than re-deriving it from scratch.

**Decision 3 — a discovered baseline failure is quarantined only if it
traces to the reorder, not merely because it lives in `robot_loop.cpp`.**
- *Context*: 3 of the 4 baseline failures (`test_plant.py`, `test_sim_api.
  py`, `test_profiled_motion_sim.py`'s turn scenario) do not obviously
  share a root cause with the 4th (`test_app_robot_loop.py`), even though
  the dispatching brief's own framing suggested the reorder as a likely
  cause for at least one. This sprint's own pre-planning repro run
  (`uv run pytest` against all 4, captured verbatim) found:
  - `test_plant.py`: `plant_harness.cpp` links `WheelPlant`/`OtosPlant`/
    `Odometry`/`NezhaMotor`/`Otos`/`BodyKinematics` directly — it never
    compiles or links `robot_loop.cpp`/`pilot.cpp` at all, so it
    STRUCTURALLY cannot be coupled to the `robot_loop.cpp` cycle reorder.
    Its failure ("OtosPlant's simulated heading tracks Odometry's own
    heading closely... expected 2.34615, got 1.65345") is a real
    plant/kinematics inconsistency from the motor-interface refactor.
  - `test_sim_api.py`: fails a hardcoded timing assertion
    (`kPace=28ms`/`virtualCycleMillis=40ms`) that assumes the OLD
    `kCycle=40`/`kSettle=4` constants; `robot_loop.cpp`'s actual current
    constants are `kCycle=20`, `kSettle=0`, `kClear=0` — a value the
    driving issue's own §5 corrections section independently confirms
    ("Cycle time is 20ms, not 50ms... robot_loop.cpp:25 now has kCycle =
    20"). This is a stale hardcoded test expectation, not a coupling to
    the live reorder experiment (the reorder is about call ORDER, not the
    `kCycle` VALUE).
  - `test_app_robot_loop.py`: fails a `ScriptedI2CBus` "no script
    under-run: motor/otos (cycles)" check in exactly the two scenarios
    that exercise `cycle()` (not `boot()`) — `robot_loop.cpp`'s own
    in-code comment ("NOTE! These requests and collects have been
    reordered for testing and development and will need to be reverted...
    before running on hardware") is the direct, named cause: the
    harness's scripted bus-transaction sequence was authored against the
    pre-reorder request/collect order.
  - `test_profiled_motion_sim.py`'s turn scenario: fails a cruise-plateau
    shape assertion against a wheel-velocity trace that visibly
    oscillates between roughly half and full speed almost every sample —
    the SAME symptom signature as a stale/alternating encoder read, and
    this harness DOES compile and link the full `robot_loop.cpp`/
    `pilot.cpp` graph, so it is a plausible (not yet certain) match for
    reorder coupling.
- *Alternatives considered*: (a) require ticket 002 to independently
  verify each failure's root cause against the ACTUAL reorder (evidence:
  does the failing harness link `robot_loop.cpp`; does the failure
  pattern match a bus-transaction-order or stale-read signature) before
  quarantining, quarantining only what is demonstrated, not assumed
  (chosen); (b) quarantine all 4 as "probably WIP-related" without
  per-test justification, to save ticket time.
- *Why this choice*: (b) risks quarantining a real, fixable regression
  (like `test_plant.py`'s plant/kinematics mismatch, which cannot possibly
  be reorder-related given it never links `robot_loop.cpp`) under a
  convenient but false "it's just WIP" umbrella — exactly the kind of
  process shortcut that leaves a real bug live. (a) costs one
  investigation step per test but produces a trustworthy, individually
  justified disposition.
- *Consequences*: ticket 002 is written with this sprint's own repro
  evidence as a starting hypothesis per test, but still requires the
  executing programmer to confirm (not just assume) before fixing or
  quarantining — see that ticket's own acceptance criteria.

### Migration Concerns

**Wire compatibility.** `PlannerConfigPatch` is a proto3 message with
`optional` (explicit-presence) fields; removing a field and `reserved`-ing
its number is a standard proto3-compatible removal (an old host binary
that still sends the now-removed field number would have it silently
ignored by `skipField()`'s forward-compatible unknown-field handling,
`messages/DESIGN.md`'s own documented behavior — not a wire break). No
host-side (`src/host/robot_radio/`) caller sets any of the 18 fields today
(grep confirmed zero matches under `src/scripts`/`src/host`/`data` for any
of the 18 field names) — this is a pure removal with no host-side
follow-up required.

**No data migration** — no persisted/on-disk data model changes; boot
config baking (`gen_boot_config.py`) already leaves `arrive_tol`/
`turn_in_place_gate` unset today (confirmed via `boot_config.cpp`'s own
comment), so removing the fields changes nothing about what boot config
produces for the fields that remain.

**No deployment sequencing concern** — sim-only this sprint; nothing here
requires a coordinated host/firmware rollout.

### SUC-002: No post-completion creep on natural Move completion
Parent: (arc-level; implements the driving issue's Step 1)

- **Actor**: host (sends a Move), firmware (`App::Pilot`)
- **Preconditions**: a Move command is active (`Motion::State != kIdle`).
- **Main Flow**:
  1. The command reaches its own completion criterion (distance/dwell/
     TIMED deadline) inside a single `Motion::Executor::tick()` call.
  2. `App::Pilot::tick()`, having captured `executor_.state()` before
     calling `executor_.tick()`, observes the before-state was non-idle
     and the after-state is `kIdle`.
  3. `Pilot` stages `Drive::setTwist(0, 0)` exactly once.
  4. On a DIFFERENT cycle where a raw `TWIST` preempts an active command,
     `RobotLoop::handleTwist()` calls `Pilot::flush()` before
     `Pilot::tick()` runs that same cycle — `stateBefore` is already
     `kIdle` by the time `tick()` samples it, so step 3's branch is never
     taken and the raw twist (already staged by `handleTwist()`) survives
     untouched.
- **Postconditions**: no robot motion after a natural completion beyond
  the cycle the completion is detected; a same-cycle raw TWIST is
  unaffected.
- **Acceptance Criteria**:
  - [ ] `Pilot::tick()` captures `Motion::State` before and after
        `executor_.tick()`.
  - [ ] `drive_.setTwist(0, 0)` is staged exactly once on a
        non-idle-to-idle transition detected inside `tick()`.
  - [ ] A same-cycle flush-then-raw-twist is unaffected (existing TWIST/
        STOP preemption behavior is not regressed).
  - [ ] SUC-001's "no nonzero command survives past the terminal zero"
        harness assertion flips from `xfail` to passing.
