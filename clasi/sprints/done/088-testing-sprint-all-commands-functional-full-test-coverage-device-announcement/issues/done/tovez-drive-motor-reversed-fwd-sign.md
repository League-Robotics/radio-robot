# Straight-drive commands spin the two wheels opposite ways â€” one drive motor is mounted backwards and needs its `fwd_sign` flipped

## Context

Stakeholder HITL observation (2026-07-07, on the stand): issuing `D`, `T`, or `S`
with equal left/right arguments makes the two drive wheels turn in **opposite**
directions instead of both driving forward. One drive motor is physically mounted
mirror-image (backwards) relative to the other, so its shaft "forward" is the
robot's "reverse." A straight command therefore yaws the robot / fights itself
instead of going straight.

## Root cause

The per-motor forward-sign knob exists and is applied â€” `msg::MotorConfig::fwd_sign`
([motor.h:88](source/messages/motor.h#L88)), applied in `NezhaMotor` where
"positive written = logical forward" ([nezha_motor.cpp:357](source/hal/nezha/nezha_motor.cpp#L357))
and in the encoderâ†’travel sign ([nezha_motor.cpp:198](source/hal/nezha/nezha_motor.cpp#L198)).
But boot config sets **`fwd_sign = 1` for every port**, including both drive-pair
ports (1 and 2):

```cpp
// boot_config.cpp:28-36  (AUTO-GENERATED)
for (uint32_t i = 0; i < kMotorConfigCount; ++i) {
    out[i] = msg::MotorConfig();
    out[i].setPort(i + 1);
    out[i].setFwdSign(1);          // <-- every port +1
    ...
}
```

For a mirror-mounted drive pair, ports 1 and 2 need **opposite** `fwd_sign` (one
`+1`, one `-1`) so that "logical forward" drives both wheels forward in the robot
frame (and so both encoders count positive when driving forward).

`source/config/boot_config.cpp` is **auto-generated** â€” "do not edit by hand.
Regenerated by `scripts/gen_boot_config.py` â€¦ Source: `data/robots/tovez_nocal.json`."
So the fix belongs in the generator and the robot JSON source of truth, not the
`.cpp`.

  per-motor forward-direction field for the drive pair, and have
  `scripts/gen_boot_config.py` bake the correct per-port `fwd_sign` (not a blanket

  signs for the pair â€” decide which layer owns wheel polarity (motor vs drivetrain)
  and keep it single-sourced.
- **Which** drive port (1 or 2) is the backwards one is a bench fact â€” determine
  on the stand and bake that port's `fwd_sign = -1`.
- Confirm the 087 greenfield config path (`Config::defaultMotorConfigs()` â†’
  `NezhaHardware`, and any connect-time calibration push via the `Configurator`)
  actually carries the sign to the motor.
