# Host codec mirror: grpc_tools.protoc pb2 codegen + serial_conn.py * branch + rogo binary send

## Description

Give the host the identical schema ticket 001 declared (compiled Python
bindings via `grpc_tools.protoc`, the SAME tool `gen_messages.py` already
uses on the firmware side) and a receive/send path for the binary plane
over the exact transport/demux the text plane already uses. This ticket
is sequenced right after 001 (not at the end) because ticket 006's
differential test harness needs `host/robot_radio/robot/pb2/` as its
`google.protobuf` reference implementation.

1. **`grpc_tools.protoc --python_out=host/robot_radio/robot/pb2`** wired
   into `justfile`/`build.py` beside the existing `gen_messages.py`
   invocation — same `protos/*.proto` source, same `grpc_tools`
   dependency already in `pyproject.toml`'s codegen group (verified:
   `grpcio-tools>=1.60,<2.0` already present). Device tables and host pb2
   bindings can never skew independently since both compile from the
   identical proto source on every build.
2. **`host/robot_radio/io/serial_conn.py`'s `_reader_loop`** gains ONE new
   branch, inserted before the final "anything else: drop silently"
   fallthrough: a line starting with `*` is dearmored (strip `*B` prefix,
   base64-decode), parsed via `pb2.ReplyEnvelope.FromString(...)`, and
   routed through the SAME `_reply_queues`/`_tlm_queue` machinery the
   existing `OK`/`ERR`/`CFG`/`ID`/`TLM` branches already use (keyed by
   `corr_id`, same `_CORR_ID_RE`-equivalent lookup — the envelope's
   `corr_id` field replaces the text plane's `#<digits>` suffix
   extraction, same semantics). **Every existing branch
   (`TLM`/`EVT`/`OK`/`ERR`/`CFG`/`ID`/`#`-comment/`keepalive`) is
   untouched** — verify by diff, not by inspection alone.
3. **`SerialConnection.send_envelope(envelope: pb2.CommandEnvelope,
   read_timeout=500, ...)`** alongside the existing text `send()` —
   serializes, base64-armors (`*B<base64>\n`), writes, and blocks on the
   corr-id-keyed reply queue exactly like `send()` does today (reuse
   `send()`'s retry-on-corrupted-command logic if it transfers cleanly;
   note in completion notes if it doesn't and a simpler non-retrying
   version was shipped instead).
4. **`host/robot_radio/io/cli.py` (`rogo`)** gains a binary send path
   (e.g. a `--binary`/`-b` flag or a dedicated subcommand) that can build
   and send a `CommandEnvelope` for each of this sprint's seven
   implemented arms (drive/segment/replace/stop/ping/echo/id).

**pb2 import packaging (the flat-import gotcha)**: confirmed by direct
experiment that `grpc_tools.protoc --python_out` run flat (`-I protos`,
matching `gen_messages.py`'s own `-I` setup, no nested output tree) emits
each `*_pb2.py`'s cross-file references as a bare top-level import (e.g.
`envelope_pb2.py` contains `import drivetrain_pb2 as drivetrain__pb2`),
which does not resolve merely because a caller does `from
robot_radio.robot.pb2 import envelope_pb2` (Python 3 has no implicit
relative-package import). Fixed with the standard/conventional shim:
`scripts/gen_pb2.py` writes `host/robot_radio/robot/pb2/__init__.py`
itself, and that `__init__.py` inserts its own directory onto `sys.path`
once, at package-import time (runs before any `*_pb2` submodule loads).
`test_envelope_pb2_cross_file_import_resolves` specifically exercises a
cross-file field (`envelope.drive` -> `DrivetrainCommand`, defined in
`drivetrain.proto`) to prove the shim actually works, not just that
`envelope_pb2`'s own locally-defined messages parse.

Also discovered and worked around (undocumented by the ticket, flagged
here): a **second**, independent circular-import hazard — importing
`robot_radio.robot.pb2.envelope_pb2` from the top of `serial_conn.py`
would be circular, because `robot_radio.robot`'s own `__init__.py`
imports `robot_radio.robot.protocol`, which imports `SerialConnection`
from `serial_conn.py` itself. Fixed with a lazy, cached import
(`_get_envelope_pb2()`, called only from inside methods, after all
modules have finished loading) plus a `TYPE_CHECKING`-guarded import for
the type hint. `cli.py` has no such hazard (nothing under
`robot_radio.robot` imports `robot_radio.io.cli`), so its `envelope_pb2`
import is a plain top-level import.

**`send_envelope()` retry logic**: did NOT reuse `send()`'s
corrupted-command retry. That retry keys off a TEXT reply's literal
`"ERR unknown"` substring, which signals "the relay's RAW250 framing
merged a keepalive with this command, and the robot rejected the
garbled text" — a text-plane-specific corruption signal. The binary
plane has no equivalent defined yet: a corrupted/malformed binary line
simply fails to decode server-side and produces no reply at all, so
there is nothing to pattern-match against and retry without a NAK the
schema doesn't (yet) define. Shipped as a single-attempt send, per the
ticket's own instruction to flag this rather than force a fit.

**pb2/ commit policy**: mirrored `source/messages/*.h` (firmware
generated headers, committed) — no `.gitignore` entry excludes
`*_pb2.py` or `pb2/`, so `host/robot_radio/robot/pb2/` is committed too.
Regenerating via `just build-sim` reproduces byte-identical
`source/messages/*.h` (confirmed: `git status` shows no diff after a
full `build-sim` run) and a functionally identical `pb2/` tree.
