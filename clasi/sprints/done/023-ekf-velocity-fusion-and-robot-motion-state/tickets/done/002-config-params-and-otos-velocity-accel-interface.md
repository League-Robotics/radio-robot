# T002: Config params and OTOS velocity/accel interface

## Description

Add four new EKF noise tuning parameters to `Config.h` and the
`TLM_FIELD_TWIST` bitmask. Extend the `IOtosSensor` interface and
`OtosSensor` concrete class with `readVelocityTransformed()` and
`readAccelTransformed()` methods. Regenerate `DefaultConfig.cpp`.

This ticket can be worked in parallel with T001 since neither depends on
the other.

Key design decisions:
- `readVelocityTransformed()` returns an `OtosVelocity { float v_mmps; float omega_rads; }` struct.
- `readAccelTransformed()` returns an `OtosAccel { float ax_mmps2; float ay_mmps2; }` struct.
- Both structs are defined in `IOtosSensor.h` alongside `OtosPose`.
- `OtosSensor::getVelocityRaw()` already exists (reads `REG_VELOCITY_XL = 0x26`).
  `readVelocityTransformed()` wraps it with the same scale and transform logic
  as `readTransformed()`.
- The OTOS acceleration register is `REG_ACCELERATION_XL = 0x2C` (per OTOS
  datasheet). `readAccelTransformed()` uses `readXYH()` on this register.
- Body-frame v derivation from OTOS velocity registers: after applying the
  mounting rotation, `v_mmps = vx_world_mmps` (forward-axis projection after
  rotation into body frame). The OTOS heading channel `h` from the velocity
  registers gives `omega_rads` directly after scale conversion. This convention
  must be documented in the implementation comment.
- LSB scale factor: verify against the OTOS datasheet. The position registers
  use 0.1 mm/LSB (int16) and heading uses 0.01 deg/LSB. The velocity registers
  use the same LSB resolution (mm/s and deg/s per LSB). Confirm this is correct.

## Implementation Plan

### Approach

Four independent changes, all additive:

1. Add fields to `Config.h` — straightforward struct extension following the
   sprint-022 pattern for `ekfQxy`/`ekfQtheta`/`ekfROtosXy`.

2. Extend `IOtosSensor.h` with two new structs and two pure-virtual methods.
   Place the struct definitions near `OtosPose` at the top of the file.

3. Extend `OtosSensor` — the concrete class already has `getVelocityRaw()` which
   reads `REG_VELOCITY_XL`. `readVelocityTransformed()` is essentially
   `readTransformed()` applied to the velocity register instead of the position
   register. The implementation can copy the transform logic from
   `readTransformed()` (which lives in `OtosSensor.cpp`) and substitute the
   velocity register. For acceleration, substitute `REG_ACCELERATION_XL = 0x2C`.

4. Regenerate `DefaultConfig.cpp` using the script.

**MockOtosSensor update:** The mock OTOS sensor used by host tests
(`host_tests/firmware.py` or a C++ mock) must also implement the two new pure-
virtual methods. The simplest implementation returns `OtosVelocity{0,0}` and
`OtosAccel{0,0}`. Verify which file contains the mock and update it; the firmware
will not link for host tests otherwise.

### Files to modify

- `source/hal/IOtosSensor.h` — add structs and pure-virtual declarations
- `source/hal/OtosSensor.h` — add `REG_ACCELERATION_XL`; declare new methods
- `source/hal/OtosSensor.cpp` — implement `readVelocityTransformed()` and
  `readAccelTransformed()`
- `source/types/Config.h` — add four fields and `TLM_FIELD_TWIST`
- `scripts/gen_default_config.py` — add four new EKF fields
- `source/robot/DefaultConfig.cpp` — regenerate (run the script)
- Host-test mock (whichever file mocks `IOtosSensor`) — add stub implementations

### Testing plan

```
python3 build.py
uv run --with pytest python -m pytest -v
```

The full pytest suite must pass without regressions. The new HAL methods are not
directly unit-tested at this stage — they are exercised via T003/T004 (firmware
integration) and T006 (replay harness and golden vectors).

### Documentation updates

Add a comment block in `OtosSensor.cpp` above `readVelocityTransformed()`
documenting: (a) the LSB scale factor used and its datasheet reference; (b) the
body-frame v derivation convention (forward-axis projection after mounting
rotation).
